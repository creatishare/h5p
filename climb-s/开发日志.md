## 当前功能概况（2026-05-21 大改造后）

一个面向少儿编程（8–12 岁）的"爬楼梯递推"交互操作题：

  没有「选关」、没有「设置」、没有「练习编号」。
- **主区采用左右两栏**（≤1100px 自动堆叠）：
  - **左栏 ~29%**：走法记录表（含 `n / f(n) / 状态` 三列 + 副标题 `n=? 进行中/已完成`）
  - **右栏 ~71%**：模拟舞台 + 底部操作栏
- **舞台**：左上角浮卡「走法增加」（默认隐藏，点过一种走法后出现，内含 `路径 X → Y` + 大字数值），
  左上方关卡 chip（1–4），中央居中的小核桃 + 台阶。
- **操作栏**：大号按钮（高 64px / 字号 20px / 字重 800）「走一级台阶」「走两级台阶」「重置」。
  按钮内**不显示 `+1` `+2` 图标**（与文案中的 `+1 种` 容易混淆）。
- **关卡按顺序解锁**：初始只能玩第 1 关；完成第 N 关后第 N+1 关亮起；
  解锁是单调递增的（即使 reset 当前关，已解锁的更高关不会回收）。

- **「提示」按钮**：弹出 `<dialog>`，用 4 步图示 + 文字讲解玩法（顺序解锁 → 跨步 → 走遍所有走法 → 找规律）。
- **完成弹窗**：4 关全部通关后，对话标题文案动态化为 `有 f(3)+f(2)=5 种走法` 等。

## ⛔ 红线：不要动的代码

### 1. 客户端桥接（`index.html` 的 `<head>` 内联 + `app.js` 顶部兜底）

**主战场在 `index.html` 的 `<head>` 内联 `<script>`**：

```html
<script>
  window.SendMessageToJs = function () {};
  var __setIframeUrlHandled = false;
  window.SetIframeUrl = function (url) {
    if (__setIframeUrlHandled) return;      // ① 本次 page-lifecycle 只处理一次
    __setIframeUrlHandled = true;
    window.SendMessageToU3d && window.SendMessageToU3d("PageLoaded");
    window.OnIframeLoad && window.OnIframeLoad();
    if (url && url !== location.href) {     // ② 同 URL 不重复 redirect
      location.href = url;
    }
  };
  window.ShowClose = function () {};
  window.HideClose = function () {};
</script>
```

**为什么放在 `<head>` 内联**：客户端在 page-loaded 事件后会**立刻**注入桥接并调用
`SetIframeUrl`。如果这套函数还在 `<body>` 末尾的 `app.js` 里，等不到 app.js 跑完
就会抛 `TypeError: window.SetIframeUrl is not a function`，页面进入 retry/timeout。

**为什么是这个实现**（每一行都有血泪）：

| 行 | 作用 | 不这么写会怎样 |
|---|---|---|
| `if (__setIframeUrlHandled) return` | 单次握手 | 客户端在收到 `OnIframeLoad` / `PageLoaded` 信号后会**再次**触发 page-loaded 事件，再注入桥接、再调 `SetIframeUrl`，自循环 691 次（见 2026-05-22 客户端日志分析） |
| `SendMessageToU3d("PageLoaded")` | 告诉 native "H5 已加载" | 不发，客户端等 timeout 然后弹 retry |
| `OnIframeLoad()` | 告诉 native iframe 元素 ready | 同上 |
| `url !== location.href` | 同 URL 不跳 | 客户端会反复传当前 URL（带 token），无条件 `location.href = url` 会导致页面反复 reload，app.js 来不及跑 `init()` |
| `if (url)` | 容错 | 客户端偶尔传 `undefined`，赋值会让浏览器导航到字符串 "undefined" |

`app.js` 顶部保留一份**幂等兜底**（用 `window.X || function...` 写法），方便单独运行 app.js
时调试不报错。重复赋值无副作用，**不要因此把任何一份删掉**。

`init()` 末尾的 `notifyClientPageLoaded()` 调用是另一处历史遗留，跟 `SetIframeUrl` 内部那
一次不在同一帧，也不要按"看着像重复就删"的本能去清理 —— 现在的握手是靠 `__setIframeUrlHandled`
做幂等的，多调一次也安全。

### 2. 客户端 WebView 不兼容的能力（已经踩过 → 已经规避）

- ❌ **CSS `mask-image` + `data:image/svg+xml`**：客户端 WebView 不支持，会渲染成纯色色块。
  现在所有图标已经全部改为 **inline `<svg>`**，全程没有 `mask-image`。如果以后再加图标，
  请继续用 inline SVG（`fill="currentColor"` 控色），**不要再走 CSS mask**。
- ❌ **中文目录名 / 中文文件名**：`getAttribute("src")` 返回字面量、`img.src` 返回 URL-encoded
  形式，两个值在 setHeroSrc 的"跳过相同 src"比较里会被认为不同；部分服务器对中文 URL
  encode 行为也不一致。2026-05-22 已经把 `小核桃跳跃/` 改成 `hero-frames/`，**别再加中文路径的静态资源**。
- ⚠️ **`Cache-Control` 缺失**：生产服务器（47.85.84.164:8088）只发 `ETag`，不发
  `Cache-Control`/`Expires`。RAF 16ms 一帧的 setHeroSrc 节奏下，每帧重发请求会导致
  `<img>` 卡帧。现在靠 `preloadHeroFrames` 把 Image 引用挂在模块作用域 + `decode()`
  把帧塞进 in-memory image cache 来兜底，但**如果换服务器，最好给静态资源加
  `Cache-Control: public, max-age=86400`**。

### 3. 跳跃帧的预加载（`preloadHeroFrames`）

```js
const __heroFrameCache = [];
function preloadHeroFrames() {
  [HERO_IDLE, ...HERO_JUMP_FRAMES].forEach((src) => {
    const img = new Image();
    img.src = src;
    if (typeof img.decode === "function") img.decode().catch(() => {});
    __heroFrameCache.push(img);            // ← 关键：保住引用
  });
}
```

`__heroFrameCache` 的存在是为了**防 GC** —— 如果只在 forEach 里用临时 `new Image()`，
forEach 结束后那些 Image 对象会被回收，浏览器的 in-memory image cache 也跟着丢。
配合服务器不发 Cache-Control 的环境（见第 2 节），第二次跳跃就会卡帧。**不要把这个数组
"优化"掉**，它不是冗余。

## 2026-05-22 当天做了什么（部署 + 客户端嵌入修复）

把页面真正部署到生产服务器（`47.85.84.164:8088`）并嵌进客户端 WebView 后，发现两类问题。
诊断过程全部基于客户端日志 `~/Library/Logs/hetao101/WalnutCoding/Player.log`。

### 一、浏览器直接访问：第二次跳跃 sprite 卡帧

**症状**：打开页面，点「走一级」第一次有完整 13 帧动画；重置后再点，小核桃只**移动不换帧**，
卡在 idle 姿势 tiao_00 上。

**根因链**：

1. 生产服务器只发 `ETag`，没有 `Cache-Control` / `Expires` → 浏览器对 hero-frames 走启发式缓存 + 304 重验
2. 原 `preloadHeroFrames` 里 `new Image()` 是 forEach 内部的临时变量，forEach 结束就被 GC，**没有 in-memory 引用保不住 image cache**
3. 第二次跳跃时 `jumpTo` 的 RAF 每 16ms 一帧调 `setHeroSrc(frames[idx])`
4. 每个 `setAttribute('src', X)` 触发新请求，而 16ms 之内根本没法等到 X 解码完成，下一帧的 Y 已经把它顶掉
5. `<img>` 实际显示停在最后一次"碰巧解码完"的图（落地后的 tiao_00），位置（纯 JS）继续动

**修复**：[app.js:222](app.js#L222) `preloadHeroFrames` 改成保引用 + `img.decode()`，
把 14 帧 + idle 全部塞进浏览器 in-memory image cache，后续 setAttribute 命中即用。

### 二、客户端 WebView：层叠出现 4 个独立 bug，逐个剥洋葱

#### 2.1 `TypeError: window.SetIframeUrl is not a function`

**症状**：客户端打开 H5，提示「页面加载失败」，retry 也没用。

**根因**：原 `app.js` 是 `<script src="./app.js">` 放在 `<body>` 末尾，桥接函数定义在 app.js 顶部。
客户端 page-loaded 事件后**立刻**注入自己的桥接 + 调 `SetIframeUrl(...)`，这时候 app.js 还没执行完。

**修复**：把 4 个桥接函数挪进 `index.html` 的 `<head>` 内联 `<script>`，最早就位。
`app.js` 顶部保留同样赋值作为兜底（`window.X || function...`）。

#### 2.2 中文路径在 WebView 下 URL encode 不对称

**症状**：和"二次跳跃卡帧"类似，但触发条件更不稳定。

**根因**：原素材目录是 `小核桃跳跃/`。`getAttribute("src")` 返回字面量 `./小核桃跳跃/tiao_00.png`，
`img.src` 返回 URL-encoded 的 `.../%E5%B0%8F%E6%A0%B8%E6%A1%83%E8%B7%B3%E8%B7%83/...`。
`setHeroSrc` 用 `getAttribute("src") !== src` 做"跳过相同 src"判断，在部分 WebView 下两值
被规范化得不一致，导致换帧逻辑卡死。

**修复**：`git mv 小核桃跳跃 hero-frames`，同步改 [app.js:128](app.js#L128) `HERO_BASE` 和
[index.html:115](index.html#L115) hero `<img>` 的 src。

#### 2.3 同 URL `location.href = url` 导致反复 reload

**症状**：2.1 修完之后客户端能打开 H5，但页面卡在 `index.html` 的"骨架"上 ——
走法表、楼梯、关卡 chip 全是空的，按钮点了没反应。

**根因**：原 `SetIframeUrl` 永远 `location.href = url`。客户端在每次 page-loaded 后都会
再调一遍 `SetIframeUrl(targetUrl)`，targetUrl 就是「当前页面 + token」。redirect 后新页面
再次触发 page-loaded → 客户端又调 → 又 redirect → app.js 始终来不及跑完 `init()`。

**修复**：加 `url !== location.href` 守卫，同 URL 不再赋值。

#### 2.4 SendMessageToU3d / OnIframeLoad 触发 native re-fire page-loaded

**症状**：2.3 修完之后**仍然**有循环。客户端日志显示 **691 次 page-loaded 周期**。

**根因**：循环不是由 `location.href = url` 驱动的（同 URL 守卫已经截住），而是由我们
**主动调用** `SendMessageToU3d("PageLoaded")` 和 `OnIframeLoad()` 驱动的 —— 这两个
回调本质是告诉 native "H5 加载完成了"，native 收到后会**再次**派发 page-loaded 事件，
然后再注入桥接、再调 SetIframeUrl，我们再发回调……自循环。

**修复**：加 `__setIframeUrlHandled` 闭包标记，本次 page-lifecycle 只处理首次调用，
第二次起直接 `return`。跳转后新页面会有新的标记位，所以「首次跳转 → 落到带 token
的页面后再握手一次」这个流程不受影响，只是从第三次客户端再叫的时候就 no-op。

### 经验教训

1. **客户端 WebView 的 page-loaded 事件比想象中早**，桥接必须在 `<head>` 内联，不能依赖外部 script。
2. **客户端的 native 回调可能跟我们的 JS 调用形成反馈环**，所有"通知 native"的 API 都要做幂等。
3. **WebView 的 URL 比较没有标准化**，写 `if (url !== location.href)` 这种比较时要意识到不同浏览器内核行为可能微妙差异。
4. **静态资源的 Cache-Control 比想象的重要**，没缓存头时 RAF 高频换帧会出问题。
5. **`new Image()` 预加载必须保引用**，否则 GC 之后浏览器 image cache 也会丢。
6. **诊断客户端嵌入问题的关键日志**：`~/Library/Logs/hetao101/WalnutCoding/Player.log`，
   搜 `SoH5` / `SetIframeUrl` / `H5Element is loaded` / `page loaded`。

## 关键架构决定（为什么这样写）

### 跳跃动画：单 RAF 时钟驱动一切

**早期试过的三种错误做法**（都被推翻）：

| 做法 | 为什么失败 |
|---|---|
| CSS transition 平移 + JS 切两帧（idle/jump） | 只切 2 帧，等于没有序列帧动画 |
| CSS transition + JS RAF 序列帧 | 两套时钟漂移，且 cubic-bezier 回弹让"落地"和"序列帧最后一帧"对不上 |
| CSS transition + CSS @keyframes（弧线 transform） + JS RAF（序列帧） | 三套时钟，更难同步 |

**现在的做法**（`jumpTo(x1, y1, x2, y2)` 在 [app.js](app.js)）：

```js
// 所有计算都在同一个 RAF 回调里基于 t = elapsed / total 推进
const x = x1 + (x2 - x1) * t;                   // 水平线性
const yLine = y1 + (y2 - y1) * t;               // 垂直线性
const arc = arcPeak * 4 * t * (1 - t);          // 抛物线，t=0.5 处峰值
els.hero.style.setProperty('--hero-x', `${x}px`);
els.hero.style.setProperty('--hero-y', `${yLine + arc}px`);
const idx = Math.floor((t / 0.92) * frames.length);
setHeroSrc(frames[idx]);                        // 同一个 t 推帧
```

跳跃期间 `.hero` 加上 `no-transition` 类禁用 CSS transition，落地后下一帧再去掉。
这样位移和序列帧从根上无法脱节。

### 关卡顺序解锁

`state.maxUnlocked` **单调递增**：完成第 N 关 → `maxUnlocked = max(maxUnlocked, N+1)`。
重置当前关只清 `state.solved` 和 `state.selected`，**不动 `maxUnlocked`**。
这样用户回头重玩任意一关，更高关不会被锁回去。

判定函数 `isUnlocked(level)` 在 `renderLevels` 和点击 handler 里都用了，双层保险（DOM `disabled` + JS 守门）。

### 「本步贡献」浮卡的两个特殊状态

```css
.contribution.is-hidden     { opacity: 0; transform: translateX(-10px) scale(0.95); }
.contribution.no-transition { transition: none !important; }
```

- 切关瞬间：`no-transition + is-hidden` 同时上，opacity 直接落到 0，**绕过淡出动画**，
  避免淡出 220ms 内露出下一关的占位文字（"选择一种走法后显示路径"）。
- 操作完成后：单独加 `is-hidden`（不加 `no-transition`），有平滑淡入。

### 序列帧文件名的坑

`hero-frames/` 只有 `tiao_00, tiao_02, ..., tiao_14`（**没有 `tiao_01`**）。
`HERO_JUMP_FRAMES` 数组里跳过 01；不要假设连续编号。

### 客户端嵌入的握手契约（2026-05-22 补）

客户端 WebView 打开 H5 的完整流程：

```
1. WebView.navigate(BuildURL)           // BuildURL = http://47.85.84.164:8088/?iframe=0
2. WebView 触发 page-loaded 事件
3. native 注入桥接 JS（定义 OnIframeLoad / SendMessageToU3d / CloseIframe，
   桥到 __H5ElementLoaded / __OnIFrameH5Msg / __H5ElementClose 这三个 native 函数）
4. native 调 window.SetIframeUrl(TargetUrl)   // TargetUrl = BuildURL + &token=... 等
5. 我们 SetIframeUrl 内部：
   - 首次：调 SendMessageToU3d("PageLoaded") + OnIframeLoad() 通知 native "我准备好了"，
           然后 location.href = TargetUrl 跳到带 token 的页面
   - 二次+：__setIframeUrlHandled 已经为 true，直接 return（避免死循环）
6. 新页面加载 → 重复 2-5，但因为 URL 已经是 TargetUrl，第 5 步不再 redirect
```

**关键**：客户端在收到 native 回调（`__H5ElementLoaded` / `__OnIFrameH5Msg`）后**会再次**派发
page-loaded → 重新注入桥接 → 重新调 SetIframeUrl。如果我们的 SetIframeUrl 不做幂等，每次
被叫都给 native 发回调，就会形成正反馈环。**真实日志里曾经跑出 691 次循环**。

## 兼容性核查清单（部署到客户端测试时重点关注）

按下文从上到下排查；客户端不通就是这一项。

### 必查项

| 能力 | 用到的地方 | 出问题的兜底方案 |
|---|---|---|
| Inline SVG | 所有图标 | 几乎所有 WebView 都支持，没问题就别动 |
| `<dialog>` + `showModal()` | 「提示」弹窗、完成弹窗 | 老 Chromium < 37 不支持。可换成普通 `<div role="dialog">` + 自己加遮罩 |
| `<dialog>::backdrop` | 弹窗背景遮罩 | 同上，老 WebView 没遮罩；可改成弹窗自带半透明背景 div |
| CSS 自定义属性 `var(--*)` | `--hero-x / --hero-y / --hero-jump-dur` | Chrome 49+ / 几乎所有现代 WebView。**不支持的话整套动画都跑不起来** |
| `requestAnimationFrame` | 跳跃动画核心 | 通用 |
| `performance.now()` | 跳跃动画 elapsed 计算 | 通用 |
| `HTMLImageElement.prototype.decode()` | preloadHeroFrames 把帧塞进 image cache | 老 WebView 没有 decode()，已经用 `typeof img.decode === "function"` 守护，没有就退化为单纯 `new Image() + src` |
| Web Audio API（合成音效，无外部文件） | `getAudioContext()`, `playTone()` | 老安卓可能不支持；首次播放必须在用户手势后 |
| ~~Chinese UTF-8 路径~~ | ~~hero 图片~~ | ✅ 2026-05-22 已规避：`小核桃跳跃/` → `hero-frames/`。**今后别再加中文路径资源** |

### 不那么关键但也要看

- 顶栏「提示」`<dialog>` 的 ESC 关闭：原生 `<dialog>` 支持，但 WebView 屏蔽硬件键的话可能用不到，问题不大
- `localStorage` 目前没用，所以刷新会丢进度，但本来设计上就没要求持久化
- `pointer-events`、`cursor: not-allowed` 这种 CSS 在客户端可能没视觉反馈，但功能上 `disabled` 属性会真正屏蔽点击

### 触屏 / 客户端 webview 特有注意

- 所有按钮都是 `<button>`，自带点击事件
- 关卡未解锁时既有 `disabled` 属性也有 JS 守门，安全
- 跳跃动画期间 `pointer-events: none` 在 `.hero` 上，避免角色挡按钮
